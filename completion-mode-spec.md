# Completion mode spec

When orchestration reaches a terminal status, the live dashboard must enter **completion mode**: keep showing the same information users need to inspect (header, tasks, events) until they dismiss the session, and make non-TTY output end in a clear final frame instead of a timed teardown that feels like a silent exit. Completion mode applies to **every** terminal orchestration status already defined as terminal in code (`completed`, `failed`, and `stopped`), using the same `TERMINAL_STATES` set in `src/dashboard.ts` and `src/tui/live-loop.ts`, so success and failure paths behave consistently. In this iteration, **in-TUI display of `summary.md` is out of scope** (no new panel or blessed box required); `RepoStoreClient.readFile(runId, "summary.md")` remains available for a later follow-up. The TTY status bar in completion mode must show **status-specific copy plus an explicit dismiss hint**: for `completed`, `Completed — Press q or Ctrl+C to exit`; for `failed`, `Failed — Press q or Ctrl+C to exit`; for `stopped`, `Stopped — Press q or Ctrl+C to exit`. The inline (`renderLiveInline`) path must, after the last poll that observes a terminal status, **emit one final static snapshot** (the same rendered snapshot content as the last live frame, without an infinite spinner or progress animation), then **end with a newline** after that output and exit without waiting for interactive input; it must not rely on repeated `console.clear()` loops that obscure the final state or leave a blank screen.

## Acceptance criteria

- When `state.status` is `completed`, `failed`, or `stopped`, the TTY dashboard **does not** auto-destroy the screen solely because the run finished; it stays up until the user presses `q` or `Ctrl+C` (same as existing layout bindings), with the final task table and event log still visible.
- While in completion mode on TTY, the status bar shows the exact dismiss strings: `Completed — Press q or Ctrl+C to exit`, `Failed — Press q or Ctrl+C to exit`, or `Stopped — Press q or Ctrl+C to exit` matching the current `state.status`.
- Polling/refresh behavior during the non-terminal **running** phase is unchanged (e.g. `POLL_INTERVAL`, manual `r` refresh) until the first observation of a terminal status, after which the UI is static except for resize handling and key handling for quit.
- **No** blessed panel, box, or dedicated region is required to render `summary.md` in this iteration; implementers may skip `3-2` unless a follow-up spec adds it.
- For non-TTY `renderLiveInline`, after a terminal status is observed there is **no** infinite spinner or ongoing “live” animation; the process prints **one** final snapshot (consistent with the last meaningful dashboard render) and a **trailing newline**, then exits without blocking on stdin.
- Non-TTY exit does not depend on a fixed multi-second delay alone to communicate completion; the user-visible result is the final snapshot text, not an empty or cleared screen.
